# Project Name: Sterlin+

## 1. Vision
The user journey transforms a casual visitor into a captivated buyer.
They arrive on a dark, cinematic page that feels like stepping into a high-end jewellery boutique.
A 3D gold ring floats in the hero, inviting exploration. As they scroll, they discover craftsmanship,
fall in love with handpicked collections, feel the desire to own something extraordinary, and finally
— with confidence — place their piece in the cart. The transformation: from **curiosity → desire → ownership**.

## 3. Frontend Architecture

**Framework:** Next.js 16 (App Router)
**State Management:** Zustand with `persist` middleware (cart state in localStorage)
**Animation Library:** GSAP 3 + ScrollTrigger plugin
**3D Engine:** React Three Fiber (`@react-three/fiber`) + Drei helpers
**Routing:** Next.js App Router (`/app` directory)
**SEO Strategy:** Metadata API in `layout.js`, semantic HTML, descriptive titles

```
/src
  /app
    layout.js          ← Root layout: Navbar + CartDrawer + grain overlay
    page.js            ← Homepage: Hero + Scene2/3/4
    globals.css        ← Design tokens, typography, scrollbar, utilities
    page.module.css    ← Hero CSS
    /products
      page.js          ← Product listing with sidebar filter (client component)
      page.module.css
      /[id]
        page.js        ← Product detail: 3D viewer + Add to Cart
        page.module.css
    /api
      /products
        route.js       ← GET all products (with category/material/featured filters)
        /[id]

  /components
    /ui
      Navbar.jsx       ← Logo, nav links, cart icon with count badge
      CartDrawer.jsx   ← Sliding cart panel with quantity controls

  /lib

  schema.prisma        ← Prisma schema (SQLite, Product model)
```

## 4. Backend Architecture

**Server:** Next.js API Routes (serverless, runs on Node.js edge)
**Database:** SQLite via `better-sqlite3` (file: `prisma/dev.db`)
**Authentication:** `next-auth` v4 installed, not yet configured
**API Structure:**

```
/api
  /products
    GET  /?category=&material=&featured=   ← list products with filters
    /[id]
      GET  /                               ← single product details
  /auth (planned – next-auth)
  /payments (planned)
```

**Security:**
- Input validation via query param checks in route handlers
- `better-sqlite3` prepared statements (SQL injection safe)
- `readonly: true` on DB connection at runtime (write only during seed)

## 5. Data Flow Diagram

```
User Scroll → GSAP ScrollTrigger → Animate DOM / Canvas elements
User visits /products → ProductsPage (client component) → fetch('/api/products')
  → Next.js Route Handler → better-sqlite3 → SELECT * FROM Product
  → JSON response → React state → Render product cards

User clicks product → /products/[id] → fetch('/api/products/:id')
  → Route Handler → better-sqlite3 → SELECT * FROM Product WHERE id = ?
  → JSON response → Render 3D viewer + product info

User clicks "ADD TO CART" → Zustand addItem() → localStorage persist
  → Navbar updates cart count badge (via getCartCount())
  → CartDrawer shows item list (via useCart store)
```

## 6. Performance Plan ✅ (Implemented)

### Lazy-Loading 3D Scenes
Three.js / React Three Fiber (~2 MB) is **never bundled into the initial page load**. Both 3D components are extracted into standalone files and loaded asynchronously:

| Component | File | Strategy |
|-----------|------|----------|
| Hero ring | `components/ui/HeroCanvas.jsx` | `dynamic(() => import(...), { ssr: false })` |
| Product viewer | `components/ui/ModelViewer.jsx` | `dynamic(() => import(...), { ssr: false })` |

- Text and HTML render **instantly** — 3D loads after hydration
- Loading placeholder: `null` on hero (invisible), text on product detail

### Code Splitting
- Next.js App Router **automatically** splits each route into its own JS chunk
- `dynamic()` imports create **additional chunks** for Three.js — never downloaded unless the user visits that page
- Result: homepage shell is lightweight; 3D downloads in the background

### Responsive Design
Breakpoints implemented in **all 10 CSS modules**:

| Breakpoint | Behaviour |
|-----------|-----------|
| `≤ 1024px` (Tablet) | Reduced font sizes, smaller image columns, sidebar narrows |
| `≤ 768px` (Mobile-L) | Single-column layouts, nav links hidden, panels stack vertically |
| `≤ 480px` (Mobile-S) | Minimum font sizes, full-width buttons, compact cart |

### Build Verification
- `npm run build` → **Exit code 0 ✅** (Turbopack, Next.js 16.1.6)
- All pages compile cleanly — no TypeScript or ESLint errors blocking the build

### Other
- **Model format**: Procedural geometry (no external GLB files) — zero 3D asset download cost
- **DB connection**: `better-sqlite3` synchronous reads are ~10ms — no async overhead on API routes
- **CDN**: Planned for production via Vercel Edge Network

## 8. Scaling Plan

**If 10K users:**
- Current SQLite setup handles this well for read-heavy jewellery catalogue
- Add basic caching headers to API routes (`Cache-Control: s-maxage=60`)

**If 100K users:**
- Migrate DB to Turso (distributed SQLite) or PlanetScale
- Add Redis caching layer for product listings
- Use Next.js `fetch` with `revalidate` for ISR (Incremental Static Regeneration)
- Implement image CDN (Cloudinary or Vercel Image Optimization)

**If 1M users:**
- Full microservices or separate product/cart/auth APIs
- Postgres (via Neon/Supabase) for ACID compliance
- Edge caching for product pages (Vercel Edge Middleware)
- Real-time inventory via WebSockets / Server-Sent Events

**Caching:**
- Product list: CDN + stale-while-revalidate headers
- Product detail: ISR with 5-min revalidation

**Load balancing:** Handled by Vercel's edge infrastructure (auto)

**Database indexing:**
```sql
CREATE INDEX IF NOT EXISTS idx_product_category ON Product(category);
CREATE INDEX IF NOT EXISTS idx_product_featured  ON Product(featured);
CREATE INDEX IF NOT EXISTS idx_product_created   ON Product(createdAt DESC);
```
